# 美妆电商客服 Agent——RAG 检索专项测评报告

测评日期：2026-07-16  
测评结论：**第一阶段弱监督检索测评已完成；推荐方案达到多数相关性指标合格线，但无答案识别和端到端性能不达标，暂不建议按生产级验收。**

## 1. 执行摘要

本次测评完成了知识库快照清理、测试集构建、7 组基线实验、2 组优化回归及性能根因定位。

核心结论如下：

1. **意图增强 Query 有显著收益。** Hybrid 使用增强 Query 后，Hit@3 从 70.2% 提升至 86.0%，MRR@10 从 0.640 提升至 0.833，说明业务意图提示和关键词对口语、多轮指代问题有效。
2. **当前生产 RRF+MMR 配置过度追求多样性。** Duplicate Rate@3 虽降至 0，但 Precision@3 仅 44.4%、nDCG@5 仅 0.623，明显低于增强 Hybrid。
3. **最佳质量与多样性平衡方案为“增强 Hybrid + 语义去重”。** Hit@3 87.7%、Group Recall@3 91.2%、MRR@10 0.851、Duplicate Rate@3 为 0。
4. **Elasticsearch 本身不是主要延迟瓶颈。** ES 内部 `took` 仅 1～5ms；`urllib.request` 每次请求约 2.0～2.1s，改为持久化 `httpx.Client` 后热请求约 8～14ms。
5. **Query Embedding 是端到端首要性能瓶颈。** 当前纯 NumPy BGE 单查询平均约 1.33s、P95 约 1.78s；即使检索侧优化后 P95 为 677ms，端到端仍约需 2 秒以上。
6. **无答案识别未达标。** 推荐方案在仅 4 条测试无答案样本上的 Recall 为 50%，容易把实时库存、医疗诊断、未公开配方等知识缺口误判为“有资料可答”。

综合判断：当前系统已经具备可展示的 RAG 检索能力，但尚未达到生产级稳定性。建议先落地 P0 优化，再建设人工标注正式验收集。

相关性使用人工定义的“必要信息点组”进行弱监督分级：

- 3：覆盖全部必要信息点，可直接支持回答。

- 1：只覆盖部分主题信息。
- 0：不相关或可能误导。

报告中的 Recall 使用 **Group Recall**，表示 Top-K 对必要信息点的覆盖率。该方法适合快速回归，但没有穷举全部相关 `doc_id`，不能替代 300～600 条双人四级标注的正式测试集。

## 4. 基线方案对照

以下延迟为原 `urllib.request` 检索客户端下的测量结果，不包含 Query 向量编码时间。

| 实验 | Hit@3 | Group Recall@3 | MRR@10 | nDCG@5 | Precision@3 | Dup@3 | P95 |
|---|---:|---:|---:|---:|---:|---:|---:|
| A BM25 原始问题 | 73.7% | 81.0% | 0.624 | 0.656 | 52.6% | 19.9% | 2301ms |
| B kNN 原始问题 | 64.9% | 72.5% | 0.607 | 0.621 | 50.9% | 19.3% | 2268ms |
| C kNN 增强 Query | 84.2% | 88.3% | 0.822 | 0.820 | 72.5% | 28.7% | 2318ms |
| D Hybrid 原始问题 | 70.2% | 78.7% | 0.640 | 0.666 | 53.8% | 22.2% | 2392ms |
| E Hybrid 增强 Query | **86.0%** | **90.4%** | **0.833** | **0.843** | **74.3%** | 28.7% | 2383ms |
| F RRF 增强 Query | 84.2% | 89.5% | 0.807 | 0.785 | 71.3% | 17.0% | 2493ms |
| G 当前生产 RRF+MMR | 73.7% | 81.9% | 0.659 | 0.623 | 44.4% | **0%** | 7110ms |

### 4.1 Query 增强收益

向量侧增强相较原始 kNN：

- Hit@3：+19.3 个百分点。
- Group Recall@3：+15.8 个百分点。
- MRR@10：+0.216。
- 多轮查询 Hit@3：从 47.4% 提升至 84.2%。

Hybrid 增强相较原始 Hybrid：

- Hit@3：+15.8 个百分点。
- Group Recall@3：+11.7 个百分点。
- MRR@10：+0.192。

因此，短问题、多轮指代和口语表达需要意图与实体关键词补全。但增强 Query 会增加 BGE 序列长度，应限制提示词和关键词预算。

### 4.2 当前 MMR 的质量回退

当前生产配置把重复率从 17%～29% 降为 0，但付出了明显代价：

- 相比增强 Hybrid，Hit@3 下降 12.3 个百分点。
- Precision@3 下降 29.9 个百分点。
- nDCG@5 下降 0.220。
- 多轮查询 Hit@3 仅 57.9%。
- P95 增加至 7.11s。

典型表现是为追求候选差异，选择了主题不同但无法回答问题的文档。MMR 不应作为所有查询的默认步骤。

## 5. 优化回归

在不修改生产代码的前提下，测评脚本模拟了：

1. 使用持久化 `httpx.Client` 连接池代替逐请求 `urllib.request`。
2. 在增强 Hybrid 或 RRF 候选上执行语义近重复过滤，但不再执行激进的 MMR 重排。

| 优化方案 | Hit@3 | Group Recall@3 | MRR@10 | nDCG@5 | Precision@3 | Dup@3 | 检索P95 |
|---|---:|---:|---:|---:|---:|---:|---:|
| H 增强 Hybrid + 语义去重 | **87.7%** | **91.2%** | **0.851** | **0.841** | **70.2%** | **0%** | **677ms** |
| I 生产式 RRF + 语义去重 | 80.7% | 87.1% | 0.774 | 0.725 | 62.0% | 0% | 877ms |

推荐采用 H 作为下一版候选方案。相较当前生产 G：

- Hit@3 提升 14.0 个百分点。
- Group Recall@3 提升 9.4 个百分点。
- MRR@10 提升 0.192。
- Precision@3 提升 25.7 个百分点。
- 保持 Duplicate Rate@3 为 0。
- 检索侧 P95 从 7.11s 降至 677ms。

## 6. 验收线对照

下表以方案 H 为候选。P95 为向量已生成后的检索侧耗时；当前端到端耗时还需加上约 1.3～1.8s 的 Query 编码。

| 指标 | 方案 H | 合格线 | 较好目标 | 判断 |
|---|---:|---:|---:|---|
| Hit Rate@3 | 87.7% | 85% | 92% | 合格，未达较好目标 |
| Group Recall@3 | 91.2% | 80% | 88% | 达较好目标 |
| Group Recall@5 | 91.2% | 85% | 92% | 合格，距目标 0.8pp |
| MRR@10 | 0.851 | 0.75 | 0.85 | 达较好目标 |
| nDCG@5 | 0.841 | 0.75 | 0.85 | 合格，略低于目标 |
| Precision@3 | 70.2% | 70% | 80% | 刚过合格线 |
| Duplicate Rate@3 | 0% | ≤20% | ≤10% | 达较好目标 |
| 无答案 Recall | 50% | 80% | 90% | **不合格** |
| 检索侧 P95 | 677ms | ≤800ms | ≤500ms | 合格，未达较好目标 |
| 当前端到端估算 P95 | ≥2.0s | ≤800ms | ≤500ms | **不合格** |

因此，本轮结论为：**相关性与重复控制基本合格，无答案识别和端到端性能不合格。**

## 7. 分场景表现

方案 H 按查询类型：

| 查询类型 | Hit@3 | Group Recall@3 | MRR@10 | nDCG@5 |
|---|---:|---:|---:|---:|
| 标准问法 | 89.5% | 92.1% | 0.895 | 0.856 |
| 口语与噪声 | 84.2% | 89.5% | 0.816 | 0.825 |
| 多轮指代 | 89.5% | 92.1% | 0.842 | 0.842 |

口语噪声仍是相对薄弱场景。多轮结果较好，是因为测试集为增强 Query 提供了从历史对话抽取出的关键词；线上效果仍依赖意图识别器能否稳定提取这些实体，需与意图专项测评联动验证。

## 8. 典型失败案例

### 8.1 品牌大小写导致精确商品查询零结果

三个 Almay 商品查询全部返回空列表。原因是查询侧品牌过滤值为 `Almay`，索引中的品牌字段为 `almay`，Elasticsearch `keyword/term` 过滤区分大小写。

错误类型：`metadata_mismatch`。

建议：新增带 lowercase normalizer 的 `brand_normalized`；入库和查询统一 `casefold()`；品牌过滤返回 0 条时自动降级为无品牌过滤并记录告警。

### 8.2 过敏与退货组合约束未同时满足

查询“精华抹完脸又红又痒，这都拆了还能退不”召回的 Top-5 均是普通退换货文档，只覆盖“退货”，没有覆盖“过敏/红痒”。

错误类型：`incomplete_information`、`semantic_confusion`。

建议：高风险组合问题必须进行约束覆盖检查；优先召回同时含 `safety_allergy + after_sale` 的标准 FAQ，或由 Reranker 对关键约束逐项打分。

### 8.3 评价返现属于知识缺口

三个“评价后是否返现/奖励”问题均只召回评价反馈、气味反馈和使用效果文档，知识库没有明确的评价返现规则。

错误类型：`knowledge_gap`，不是单纯检索算法问题。

建议：新增人工审核的评价活动标准 FAQ；若平台不允许诱导好评，应明确输出合规规则，而不是从普通评价案例推断。

### 8.4 无答案查询误判

方案 H 对测试无答案问题的 Recall 仅 50%。医疗诊断和未发布新品配方仍获得较高检索分数，系统缺少可靠的拒答阈值和知识范围判断。

建议：使用独立验证集联合校准 Top-1 分数、Top-1/Top-2 margin、Reranker 分数和文档权威类型；高风险领域再叠加规则判断与人工转接。

## 9. 性能根因

### 9.1 Query 编码

| 模式 | 实测 |
|---|---:|
| 纯 NumPy BGE 单查询平均 | 1326ms |
| 纯 NumPy BGE 单查询 P95 | 1782ms |
| 65 条原始 + 65 条增强 Query 批量摊销 | 1785ms/条 |

当前实现是教学版 NumPy BERT 前向，不适合在线请求和大规模评测。

### 9.2 ES 客户端

同一个 BM25 查询的对照：

| 调用方式 | 客户端耗时 | ES `took` |
|---|---:|---:|
| `urllib.request` 每次新请求 | 2029～2110ms | 1～3ms |
| PowerShell 热请求 | 17～22ms | 1～5ms |
| 持久化 `httpx.Client` 热请求 | 8～14ms | 1ms |

索引查询不是主要瓶颈，客户端连接方式才是。生产代码应使用连接池，并复用客户端实例。

## 10. 优化建议

### P0：进入下一轮验收前必须完成

1. **替换 ES HTTP 客户端。** 使用官方 Elasticsearch Python Client 或常驻 `httpx.Client`，配置连接池、Keep-Alive、超时和重试；禁止每次查询新建 `urllib` 请求链路。
2. **替换 Query Embedding 执行器。** 使用 ONNX Runtime、Sentence Transformers/PyTorch 或独立 Embedding 服务；保持 BGE 模型与向量维度不变，验证新旧向量余弦一致性。目标：CPU P95 <150ms，理想 <100ms。
3. **将默认检索改为增强 Hybrid + 语义去重。** 暂停默认 MMR；只在重复率超过阈值或需要多知识点覆盖时条件触发。候选数建议从 30 开始压测 10/20/30。
4. **统一元数据规范。** 为品牌、品类和意图建立规范化字段；查询过滤零命中时执行安全降级，避免正确内容被硬过滤。
5. **补齐无答案决策。** 扩充至少 50 条无答案样本，按高风险/普通知识缺口分别校准阈值；未达阈值时返回澄清或转人工，不进入答案生成。

### P1：提升质量与稳定性

1. 新增过敏售后、评价活动、品牌政策等人工审核的 canonical FAQ。
2. 对组合问题做关键约束覆盖打分，Reranker 必须逐项检查实体、条件、动作和范围。
3. 在入库阶段生成 `qa_fingerprint`、`answer_hash` 或重复组 ID，优先按稳定字段去重，减少在线 `SequenceMatcher` 的 O(n²) 计算。
4. 对增强 Query 设置长度预算，只保留 2～3 个区分度最高的关键词；评估“BM25增强+向量原问”和“紧凑向量增强”的差异。
5. 将 Query Embedding 缓存作为正式评测资产，检索融合调参不再重复编码。

### P2：正式生产验收

1. 建设 300～600 条人工 `doc_id` 四级相关性标注集。
2. 至少 20% 双人标注，目标加权 Kappa ≥0.80。
3. 增加品牌、地区、时效和政策版本等元数据冲突样本。
4. 在 CI 中固定基准与向量缓存，设置 Hit@3、nDCG、重复率和延迟回归门禁。
5. 增加并发 1/5/20 下的 P50/P95/P99、吞吐、错误率和资源占用压测。

## 13. 测评限制

1. 当前是第一阶段弱监督集，相关性依赖人工信息点规则，不是穷举 `doc_id` 的双人标注集。
2. 无答案测试集仅 8 条，其中测试切分 4 条，50% Recall 具有较大统计波动，但足以证明当前能力未达标。

4. 未调用 LLM Reranker，避免模型费用和非确定性；Reranker 需要在固定候选池上另做专项实验。
5. 纯业务订单查询不计入检索相关性指标。
