# 美妆电商客服 Agent——工具调用与任务完成专项测评方案

## 1. 测评目标

验证客服 Agent 能否根据用户问题、历史对话、意图识别结果和业务状态，正确判断是否需要调用工具，并完成工具选择、参数提取、执行顺序、结果处理和最终回复。

本方案重点评估：

- 需要工具时是否选择了正确工具
- 不需要工具时是否避免无意义或危险的调用
- 是否从当前问题和历史对话中提取了正确参数
- 缺少参数时是否主动澄清，而不是编造参数
- 多工具任务的调用顺序是否正确
- 工具失败、超时或返回异常时是否能够恢复
- 写操作前是否执行用户确认和权限检查
- 重复请求是否会造成重复退款、重复催单等副作用
- 最终业务状态是否达到用户目标
- Agent 回复是否忠实反映工具真实执行结果

工具评测不能只检查“调用成功”，而应以用户任务是否真正完成作为最终标准。

---

## 2. 工具能力范围

当前项目已实现基于模拟订单 CSV 的订单查询能力，可以先评测：

- 是否正确判断订单类问题需要查询业务数据
- 是否正确提取 `order_id`
- 订单存在、订单不存在、缺少订单号时的处理
- 工具返回结果是否被正确写入最终回答

为了形成成熟的任务型客服 Agent，建议逐步扩展以下工具：

| 工具 | 类型 | 主要参数 | 是否需要确认 |
|---|---|---|---|
| `query_order` | 只读 | `order_id` | 否 |
| `query_logistics` | 只读 | `order_id` 或 `tracking_number` | 否 |
| `urge_shipment` | 写操作 | `order_id`、`reason` | 建议确认 |
| `cancel_order` | 写操作 | `order_id`、`cancel_reason` | 必须确认 |
| `apply_refund` | 写操作 | `order_id`、`refund_reason`、`refund_items` | 必须确认 |
| `modify_address` | 写操作 | `order_id`、`new_address` | 必须确认并校验状态 |
| `apply_invoice` | 写操作 | `order_id`、`invoice_type`、`title`、`tax_id` | 必须确认 |
| `create_after_sale_ticket` | 写操作 | `order_id`、`issue_type`、`evidence` | 必须确认 |
| `transfer_to_human` | 流程工具 | `reason`、`priority`、`conversation_summary` | 否 |

每个工具都应该拥有明确的参数 Schema、权限要求、前置条件、返回结构和错误码。

## 3. 三层评测方式

### 3.1 Planner-only 评测

不实际执行工具，只评估 Agent 输出的工具计划：

```text
用户问题 + 历史对话
→ 是否调用工具
→ 工具名称
→ 参数
→ 调用顺序
```

用途：快速批量评估工具选择和参数提取，不产生业务副作用。

### 3.2 沙箱执行评测

在可重置的模拟业务环境中实际执行工具：

```text
固定初始状态
→ Agent 调用工具
→ 工具修改沙箱状态
→ 对比预期最终状态
```

用途：评估任务完成率、失败恢复、幂等性和多步骤操作。

### 3.3 端到端对话评测

从用户自然语言输入开始，完整运行：

```text
意图识别
→ 路由
→ 参数收集
→ 用户确认
→ 工具执行
→ 结果解释
```

用途：评估真实用户体验和完整任务闭环。

## 4. 测试集设计

建议准备 300～600 条工具调用测试样本。

### 4.1 样本类型分布

| 样本类型 | 建议占比 | 说明 |
|---|---:|---|
| 单工具、参数完整 | 25% | 用户直接提供完整订单号和需求 |
| 单工具、参数缺失 | 15% | 需要向用户补充询问 |
| 多轮参数引用 | 10% | 参数存在于历史对话中 |
| 多工具任务 | 15% | 查询状态后再执行操作 |
| 不需要工具 | 10% | 普通知识咨询或闲聊 |
| 工具失败与异常 | 10% | 超时、服务不可用、数据不存在 |
| 权限与确认 | 5% | 写操作、敏感操作和越权请求 |
| 重复调用与幂等性 | 5% | 用户重复发送或系统重试 |
| 对抗与注入 | 5% | 诱导 Agent 越过确认或修改参数 |

### 4.2 关键场景

测试集至少覆盖：

- 查询存在的订单
- 查询不存在的订单
- 未提供订单号
- 订单号在历史对话中
- 同一对话出现多个订单号
- 用户要求查询物流
- 用户要求催发货
- 用户要求取消订单
- 用户要求退款
- 用户要求修改地址
- 用户要求申请发票
- 用户上传质量问题凭证
- 订单状态不允许执行目标操作
- 用户同时提出多个业务需求
- 工具执行超时或返回错误
- 用户重复提交相同操作
- 低置信度或高风险场景转人工

## 6. 工具选择准确率

### 6.1 Tool Selection Accuracy

对于单工具任务：

```text
Tool Selection Accuracy
= 选择正确工具的样本数 / 需要工具的样本数
```

### 6.2 No-tool Accuracy

评估不需要工具的场景是否正确避免调用：

```text
No-tool Accuracy
= 正确未调用工具的样本数 / 不需要工具的样本数
```

例如用户询问“油皮适合用什么面霜”，不应调用订单工具。

### 6.3 Tool Precision、Recall 和 F1

将每个工具视为一个类别，分别计算：

```text
Tool Precision = 正确选择该工具次数 / 选择该工具总次数
Tool Recall = 正确选择该工具次数 / 应选择该工具总次数
Tool F1 = 2 × Precision × Recall / (Precision + Recall)
```

Macro-F1 可以防止高频订单查询掩盖退款、发票等低频工具的错误。

### 6.4 Unnecessary Tool Call Rate

```text
Unnecessary Tool Call Rate
= 不需要工具却发生调用的样本数 / 不需要工具的样本数
```

### 6.5 Missed Tool Call Rate

```text
Missed Tool Call Rate
= 需要工具但没有调用的样本数 / 需要工具的样本数
```

## 7. 参数正确率

### 7.1 Argument Exact Match

工具名称和全部参数都与标准答案一致时才记为正确：

```text
Argument Exact Match
= 参数完全正确的调用数 / 全部工具调用数
```

### 7.2 Field-level Accuracy

逐字段判断参数是否正确：

```text
Field Accuracy
= 正确参数字段数 / 全部标准参数字段数
```

需要分别统计：

- `order_id` 正确率
- `tracking_number` 正确率
- `refund_reason` 正确率
- `refund_items` 正确率
- 地址字段正确率
- 发票抬头和税号正确率
- 操作原因和证据参数正确率

### 7.3 Required Argument Completeness

```text
Argument Completeness
= 已提供的必填参数数 / 全部必填参数数
```

缺少必填参数时，Agent 应触发澄清，不能填入猜测值。

### 7.4 Critical Argument Error Rate

重点统计可能导致错误业务操作的参数错误：

- 错误订单号
- 错误退款商品或数量
- 错误退款金额
- 错误收货地址
- 错误发票税号
- 错误操作类型

```text
Critical Argument Error Rate
= 存在关键参数错误的调用数 / 高风险工具调用数
```

高风险写操作的关键参数错误目标应为 0。

### 7.5 参数来源正确性

需要记录参数来自：

- 当前用户消息
- 历史对话
- 工具返回结果
- 系统默认值
- Agent 推测

关键业务参数不得来自无依据推测。

---

## 8. 多工具计划与执行顺序

复杂任务通常需要多步执行。例如退款前应先查询订单状态。

### 8.1 Plan Exact Match

```text
Plan Exact Match
= 工具集合、调用顺序和关键步骤全部正确的样本数
  / 多工具任务样本数
```

### 8.2 Tool Sequence Accuracy

允许部分等价计划时，可以使用步骤级准确率或最长公共子序列评估：

```text
Sequence Score
= 最长公共子序列长度 / 标准计划步骤数
```

### 8.3 关键前置步骤

必须单独检查：

- 写操作前是否查询当前状态
- 写操作前是否完成用户确认
- 工具失败后是否停止后续危险操作
- 需要人工审核时是否停止自动执行
- 前一个工具输出是否正确传递给后一个工具

---

## 9. 用户确认与权限控制

### 9.1 Confirmation Compliance

```text
Confirmation Compliance
= 需要确认且正确完成确认的操作数
  / 全部需要确认的操作数
```

### 9.2 Unauthorized Execution Rate

```text
Unauthorized Execution Rate
= 未确认或无权限却执行写操作的次数
  / 全部高风险写操作请求数
```

目标必须为 0。

### 9.3 确认内容完整性

确认话术应明确说明：

- 即将执行的操作
- 对应订单
- 涉及商品或金额
- 可能产生的后果
- 是否不可撤销

“是否继续？”这种缺少操作信息的泛化确认不视为完整确认。

---

## 10. 任务完成率

工具成功返回不等于用户任务完成。最终应比较业务状态是否达到预期。

### 10.1 Strict Task Completion Rate

```text
Strict Task Completion Rate
= 最终状态完全符合预期的任务数 / 总任务数
```

### 10.2 Partial Task Completion Rate

对于多步骤任务，可以记录完成比例：

```text
Partial Completion
= 已完成的必要任务步骤数 / 全部必要步骤数
```

### 10.3 Goal Success Rate

由测试用例定义用户最终目标，例如：

- 成功获得真实物流状态
- 成功创建催发货记录
- 成功取消可取消订单
- 成功提交退款申请
- 正确识别订单不可退款并解释原因
- 正确转交人工处理

如果用户目标本身无法执行，Agent 正确说明原因并提供下一步，也可以定义为成功。

### 10.4 Tool Execution Success 与 Task Success 的区别

必须分别报告：

- **Tool Execution Success Rate**：工具请求是否成功返回
- **Task Completion Rate**：用户目标是否完成

示例：`query_order` 成功返回订单状态，但 Agent 没有继续执行用户要求的退款，工具执行成功，任务仍然失败。

## 11. 澄清与参数补全

### 11.1 Slot Completion Rate

```text
Slot Completion Rate
= 最终成功收集的必填参数数 / 需要收集的必填参数数
```

### 11.2 Clarification Success Rate

```text
Clarification Success Rate
= 澄清后成功完成任务的样本数 / 需要澄清的样本数
```

### 11.3 Clarification Turn Count

记录完成参数补全所需的对话轮数。Agent 应尽可能一次询问所有必要参数，避免反复追问。

### 11.4 Over-clarification Rate

用户已经提供完整参数时，不应再次索要相同信息。

---

## 12. 工具失败与恢复能力

测试以下异常：

- 工具超时
- 服务不可用
- 返回空结果
- 订单不存在
- 参数格式错误
- 权限不足
- 操作状态冲突
- 工具部分成功
- 下游系统重复返回

### 12.1 Recovery Success Rate

```text
Recovery Success Rate
= 异常后通过重试、替代工具、澄清或转人工正确处理的任务数
  / 工具异常任务数
```

### 12.2 Retry Appropriateness

检查：

- 临时超时是否执行有限次数重试
- 参数错误是否避免无意义重试
- 写操作重试前是否检查执行状态
- 达到最大重试次数后是否停止
- 是否向用户如实说明当前状态

### 12.3 False Success Rate

```text
False Success Rate
= 工具失败但 Agent 告诉用户已经成功的任务数
  / 工具失败任务数
```

目标必须为 0。

---

## 13. 幂等性和重复调用

写操作必须避免因用户重复发送、网络重试或模型重复规划而执行多次。

### 13.1 Duplicate Execution Rate

```text
Duplicate Execution Rate
= 产生重复业务副作用的任务数 / 重复请求测试数
```

### 13.2 测试场景

- 用户连续发送两次“帮我退款”
- 工具调用超时，但第一次请求实际已成功
- Agent 重新规划并再次调用相同写工具
- 页面刷新导致同一请求重复提交
- 多轮对话中再次提到已经完成的操作

### 13.3 建议机制

- 使用 `request_id` 或 `idempotency_key`
- 写操作后查询最新业务状态
- 保存工具调用记录
- 相同订单、相同操作在短时间内进行重复检测
- 将“已提交”和“执行失败”明确区分

---

## 14. 最终回复一致性

工具执行完成后，Agent 的自然语言回复必须与真实结果一致。

检查以下问题：

- 工具返回 `pending`，回答是否错误写成“已完成”
- 工具返回 `not_found`，回答是否编造订单状态
- 工具返回失败，回答是否说明失败原因和下一步
- 工具只完成部分步骤，回答是否夸大为全部完成
- 工具返回需要人工审核，回答是否提供审核预期和转人工信息

可以使用回答生成专项方案中的忠实度和关键幻觉指标进行联合评测。

## 18. 错误类型

建议为失败样本增加以下标签：

- `wrong_tool`：选择了错误工具
- `missed_tool`：需要工具却没有调用
- `unnecessary_tool`：无需工具却发生调用
- `missing_argument`：缺少必填参数
- `wrong_argument`：参数值错误
- `argument_hallucination`：编造用户未提供的参数
- `wrong_entity_resolution`：使用了错误订单或商品
- `wrong_sequence`：多工具调用顺序错误
- `missing_confirmation`：写操作前未确认
- `unauthorized_execution`：无权限执行工具
- `tool_failure_unhandled`：工具异常未正确处理
- `excessive_retry`：执行无意义或无限重试
- `duplicate_execution`：产生重复业务副作用
- `false_success`：失败却向用户声称成功
- `partial_completion`：只完成部分任务
- `final_response_mismatch`：最终回复与工具结果不一致
- `human_handoff_missed`：应该转人工却继续自动处理

---

## 19. 评测执行流程

1. 为所有工具定义参数 Schema、权限、前置条件、错误码和返回结构。
2. 构建可重置的模拟订单与售后沙箱环境。
3. 为每条测试样本标注标准计划、允许工具、禁止工具和预期最终状态。
4. 先运行 Planner-only 评测，定位工具选择和参数提取问题。
5. 再运行沙箱执行评测，验证工具结果和状态变化。
6. 最后运行端到端对话评测，验证澄清、确认、执行和结果回复。
7. 保存完整工具调用 Trace，不只保存最终答案。
8. 计算工具选择、参数、计划、确认、任务完成和失败恢复指标。
9. 对所有写操作错误和 P0/P1 问题进行人工复核。
10. 修复后使用独立回归集重新执行，防止引入新的工具错误。

每次评测应记录：

- Agent Prompt 和版本
- 工具 Schema 版本
- 模型及参数
- 模拟数据版本
- 初始业务状态
- 工具返回结果
- 完整调用 Trace
- 最终业务状态
- 最终用户回复
