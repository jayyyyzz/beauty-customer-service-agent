# 美妆电商客服 Agent——RAG 检索专项测评方案

## 1. 测评目标

验证客服 Agent 的检索系统能否针对用户问题，从知识库中召回正确、相关、完整且尽量不重复的知识片段，为后续答案生成提供可靠依据。

本方案重点评估：

- 正确知识是否出现在 Top-K 结果中
- 最相关知识是否排在靠前位置
- 多个相关知识片段的整体排序是否合理
- BM25、向量检索、混合检索的效果差异
- 意图增强 Query 是否真正提升召回效果
- 多轮上下文、口语表达、错别字和情绪化表达下的检索稳定性
- overlap、近似问法和重复数据是否造成召回结果同质化
- 检索结果是否包含足够的来源、主题和业务元数据

当前项目可以围绕以下检索链路开展评测：

```text
用户问题 + 历史对话
→ 意图与关键词增强 Query
→ BGE 向量编码
→ BM25 / kNN / Hybrid 检索
→ Top-K 知识片段
→ 后续答案生成
```

## 2. 测评范围

### 2.1 纳入评测的查询

主要评估以下路由中的知识检索部分：

- `knowledge_base`：商品、肤质、成分、功效、用法等知识类问题
- `hybrid`：售后、发票、质量问题等需要知识和业务数据共同回答的问题

### 2.2 不直接纳入检索指标的查询

纯业务查询原则上不应依赖 RAG 知识召回，例如：

- 具体订单物流位置
- 具体订单是否发货
- 具体退款进度
- 具体订单是否可取消

这类问题应进入业务工具专项评测。如果系统错误地对纯业务问题执行知识检索，可以在路由评测中记录为错误。

### 2.3 知识库版本

每次评测必须冻结知识库快照，并记录：

- Chunk 数据版本
- Chunk 数量
- Embedding 模型及版本
- 向量维度
- Elasticsearch 索引名称
- 索引 Mapping 版本
- 检索模式和权重配置
- Query 构造策略版本
- Reranker 版本（如有）

当前项目初始评测可以使用：

```text
Chunk 方案：qa_topic
Chunk 数量：2583
Embedding：bge-small-zh-v1.5
向量维度：512
检索模式：BM25 / kNN / Hybrid
```

后续接入商品、FAQ、政策、物流规则等多源数据后，需要重新生成完整评测集和基准结果。

## 3. 测试集设计

建议构建 300～600 条人工标注查询。每个查询必须至少标注一个相关文档；对于需要多条知识共同回答的问题，应标注多个相关文档。

### 3.1 查询类型分布

| 查询类型 | 建议占比 | 说明 |
|---|---:|---|
| 标准问法 | 25% | 与知识库表达较接近 |
| 同义改写 | 20% | 用不同措辞表达相同需求 |
| 口语及噪声 | 15% | 错别字、缩写、碎片消息、情绪表达 |
| 多轮上下文 | 15% | 当前问题依赖历史对话才能理解 |
| 跨知识组合 | 10% | 需要多个知识片段共同回答 |
| 易混淆查询 | 10% | 表面关键词相似但实际意图不同 |
| 无答案查询 | 5% | 知识库没有足够信息，应拒答或澄清 |

### 3.2 业务主题覆盖

测试集应覆盖当前主要知识意图：

- 商品基础信息
- 肤质适配
- 肌肤问题
- 成分咨询
- 商品功效
- 使用方法
- 护肤流程
- 搭配禁忌
- 色号妆效
- 正品与保质期
- 安全与过敏
- 商品对比
- 赠品和小样
- 评价活动
- 售后政策
- 发票规则
- 质量问题处理规则

每个一级业务主题建议至少包含 15～20 条查询。高风险主题，如过敏、产品质量和退款规则，应增加专项样本。

```json
  "relevant_docs": [
    {
      "doc_id": "qa_topic_xxxxx",
      "relevance": 3
    }

```

### 3.4 无答案查询示例

```json
{
  "query_id": "retrieval_0201",
  "history": [],
  "question": "这款精华经过了多少人的临床双盲实验？",
  "intent": "efficacy",
  "difficulty": "hard",
  "has_answer": false,
  "relevant_docs": []
}
```

无答案样本用于验证系统是否能够识别知识缺口，不能为了提高召回率而把弱相关内容当作正确答案。

## 4. 相关性标注规范

为了计算 nDCG，需要使用分级相关性，而不是只标注相关或不相关。

### 4.1 四级相关性

| 分值 | 相关性等级 | 标注标准 |
|---:|---|---|
| 3 | 高度相关 | 可以直接、完整地支持回答当前问题 |
| 2 | 相关 | 能提供重要信息，但不足以独立完成回答 |
| 1 | 弱相关 | 主题接近，仅能提供背景信息 |
| 0 | 不相关 | 无法支持回答，或可能误导模型 |

### 4.2 标注原则

- 相关性依据文档内容判断，不能只看主题标签。
- 包含相同关键词但业务含义不同的文档应标为 0。
- 内容重复的多个 Chunk 可以都标为相关，但需要额外标注重复组。
- 规则、政策、商品信息必须与用户询问的对象和条件一致。
- 涉及时间、品牌或地区限制时，过期或范围不符的文档不能标为高度相关。
- 无法确定时应由第二名标注人员复核。

### 4.3 双人标注与一致性

建议至少抽取 20% 的测试集进行双人独立标注，并计算 Cohen's Kappa 或加权 Kappa。

建议目标：

```text
Kappa ≥ 0.80：标注一致性较好
0.60 ≤ Kappa < 0.80：需要复核部分标准
Kappa < 0.60：标注规范或意图边界需要重做
```

## 6. 核心指标

## 6.1 Recall@K

Recall@K 衡量前 K 条结果覆盖了多少标准相关文档。

```text
Recall@K = Top-K 中相关文档数量 / 全部标准相关文档数量
```

建议计算：

- Recall@1
- Recall@3
- Recall@5
- Recall@10

当每个查询只有一个标准相关文档时，Recall@K 等价于该文档是否出现在 Top-K。

示例：某问题有 2 个标准相关文档，Top-3 命中其中 1 个：

```text
Recall@3 = 1 / 2 = 0.5
```

### 6.2 Hit Rate@K

衡量 Top-K 中是否至少出现一个相关文档：

```text
Hit Rate@K = Top-K 至少命中一个相关文档的查询数 / 总查询数
```

对于客服问答，Hit Rate@3 很有业务价值，因为答案生成通常只需要至少一个可靠依据。

### 6.3 Precision@K

衡量 Top-K 结果中有多少是相关内容：

```text
Precision@K = Top-K 中相关文档数量 / K
```

Precision@K 可以反映上下文污染程度。即使 Recall 很高，如果 Top-K 中包含大量无关内容，也可能导致大模型产生错误回答。

### 6.4 MRR@K

Mean Reciprocal Rank 关注第一个相关文档出现的位置。

```text
RR = 1 / 第一个相关文档的排名
MRR = 所有查询 RR 的平均值
```

示例：

| 第一个相关文档排名 | RR |
|---:|---:|
| 1 | 1.0 |
| 2 | 0.5 |
| 3 | 0.333 |
| 5 | 0.2 |
| 未命中 | 0 |

MRR 越高，说明真正有价值的知识越靠前。

建议计算 MRR@3、MRR@5 和 MRR@10。

### 6.5 nDCG@K

nDCG 用于评估多级相关性和整体排序质量，能够区分高度相关、一般相关和弱相关文档。

```text
DCG@K = Σ ((2^rel_i - 1) / log2(i + 1))
nDCG@K = DCG@K / IDCG@K
```

其中：

- `rel_i` 是第 i 条检索结果的相关性分值
- IDCG 是理想排序下的 DCG
- nDCG 取值范围为 0～1，越接近 1 越好

建议计算：

- nDCG@3
- nDCG@5
- nDCG@10

### 6.6 MAP@K（可选）

当一个查询可能对应多个相关文档时，可以补充 Mean Average Precision：

```text
AP@K = 每个相关文档命中位置上的 Precision 的平均值
MAP@K = 所有查询 AP@K 的平均值
```

MAP 可以同时衡量相关文档是否被召回以及是否排在靠前位置。

## 7. 无答案查询评测

对于知识库中没有答案的问题，不能只计算传统 Recall，需要独立评估知识缺口识别能力。

### 7.1 指标

- **No-answer Precision**：系统判定无答案时，真正无答案的比例
- **No-answer Recall**：真正无答案的问题中，被系统正确识别的比例
- **错误召回率**：无答案查询中，系统仍返回高相关结果的比例
- **知识缺口发现率**：系统是否能输出需要补充的知识类型

### 7.2 判断方式

可以基于以下信号综合判断：

- Top-1 相似度低于阈值
- Top-K 文档相关性均较低
- Reranker 分数低于阈值
- 检索文档与用户指定商品、品牌或政策范围不一致
- 答案生成模型判断现有资料不足

阈值必须通过验证集选择，不能直接使用经验值。

---

## 8. 重复召回与结果多样性

当前对话 Chunk 使用 overlap，同一段问答可能出现在多个 Chunk 中，因此需要单独测量重复召回。

### 8.1 Duplicate Rate@K

```text
Duplicate Rate@K = Top-K 中重复或高度相似结果数量 / K
```

重复可以通过以下方式识别：

- 相同 QA 文本
- 相同 `session_id` 且核心内容高度重合
- 文本相似度超过设定阈值
- 属于同一个人工标注的重复组

### 8.2 Unique Information Coverage@K

衡量 Top-K 结果包含多少独立信息点，而不是多少条文档。

例如“过敏退货”问题需要：

1. 立即停用及安全建议
2. 过敏售后规则
3. 退款或凭证要求

如果 Top-3 都是同一句退款话术，则文档数为 3，但独立信息覆盖可能只有 1。

### 8.3 优化方向

- 按 `session_id` 或重复组去重
- 使用 MMR 增加结果多样性
- 降低 overlap 内容对向量的影响
- 只向模型传入 `core_text`
- 在 Rerank 后执行语义去重

## 9. 检索方案对照实验

至少对以下方案使用同一测试集进行离线对比：

| 实验编号 | 检索方案 | Query 内容 | 向量字段 | Rerank |
|---|---|---|---|---|
| A | BM25 | 原始问题 | 无 | 无 |
| B | kNN | 原始问题 | question_vector | 无 |
| C | kNN | 意图增强 Query | question_vector | 无 |
| D | Hybrid | 原始问题 | question_vector | 无 |
| E | Hybrid | 意图增强 Query | question_vector | 无 |
| F | Hybrid | 意图增强 Query | 多路向量 | 有 |

每个实验输出：

- Recall@1/3/5/10
- Hit Rate@1/3/5
- MRR@3/5/10
- nDCG@3/5/10
- Precision@3/5
- Duplicate Rate@3/5
- 平均检索耗时和 P95 检索耗时

### 9.1 混合检索融合方式

BM25 和向量相似度的分值范围不同，直接相加可能造成某一路长期占据主导。建议对比：

- Elasticsearch 原生 boost 加权
- 分数归一化后加权
- Reciprocal Rank Fusion（RRF）
- Reranker 二阶段排序

## 13. 错误类型

建议给失败查询添加以下标签：

- `no_hit`：Top-K 中没有相关文档
- `low_rank`：相关文档存在但排序过低
- `keyword_mismatch`：BM25 无法匹配同义表达
- `semantic_confusion`：向量检索召回语义相近但业务错误的文档
- `context_missing`：未使用历史对话导致 Query 信息不足
- `intent_pollution`：错误意图或关键词污染检索 Query
- `duplicate_results`：Top-K 结果高度重复
- `metadata_mismatch`：品牌、商品、政策范围或地区不匹配
- `knowledge_gap`：知识库本身缺少答案
- `stale_knowledge`：召回过期规则或旧版本知识
- `score_fusion`：BM25 与向量分数融合不合理
- `wrong_vector_field`：检索字段与问题类型不匹配

## 14. 评测执行流程

1. 冻结测试集、知识库、Embedding 模型和检索配置。
2. 完成人工相关性标注并检查标注一致性。
3. 分别运行 BM25、kNN 和 Hybrid 基线实验。
4. 保存每条查询的完整 Top-K 结果、分数、排名和耗时。
5. 计算 Recall@K、Hit Rate@K、Precision@K、MRR、nDCG 和重复率。
6. 按意图、查询类型和难度生成切片报告。
7. 分析无召回、低排序、重复召回和知识缺口案例。
8. 对 Query 构造、融合权重、去重和 Rerank 进行优化。
9. 使用独立回归测试集验证优化效果。
10. 保存实验配置和结果，保证不同方案可复现对比。

建议固定以下随机性和配置：

- Embedding 模型版本
- Query 前缀
- Top-K
- `num_candidates`
- BM25 与向量权重
- 向量字段
- Reranker 模型及阈值
- Elasticsearch 版本
