# 美妆电商客服 Agent——性能、成本与人工转接专项测评方案

## 1. 测评目标

验证客服 Agent 在真实业务负载下能否以可接受的响应速度、稳定性和成本完成用户任务，并在自动处理不安全、不确定或无法继续时正确转接人工客服。

本方案重点评估：

- 用户从发出问题到看到回复需要等待多久
- 意图识别、检索、工具调用和回答生成分别消耗多少时间
- 冷启动、模型加载和并发请求是否造成明显延迟
- 请求超时、异常和服务降级时系统是否仍然可用
- 每次请求、每轮对话和每个成功任务消耗多少 Token 和费用
- Token 成本是否随着历史对话和检索上下文持续失控增长
- 哪些问题应该自动解决，哪些问题必须转人工
- Agent 是否存在过度转人工或漏转人工
- 转人工时是否提供完整的用户诉求、订单信息和已执行步骤
- 人工接手后是否需要重新询问用户已经提供的信息

性能、成本和人工转接不能孤立评估。为了降低转接率而让 Agent 冒险回答，或者为了节省 Token 而遗漏安全信息，都属于错误优化。

---

## 2. 测评范围

建议分别评估以下 Agent 路由：

| 路由 | 主要步骤 |
|---|---|
| `clarify` | 意图识别 → 澄清回复 |
| `knowledge_base` | 意图识别 → Query 构造 → Embedding → ES 检索 → 答案生成 |
| `business_api` | 意图识别 → 参数提取 → 业务工具 → 答案生成 |
| `hybrid` | 意图识别 → 检索 + 业务工具 → 答案生成 |
| `fallback` | 意图识别 → 兜底或转人工 |
| `human_handoff` | 判断转人工 → 生成摘要 → 创建人工工单 |

不同路由步骤数量不同，必须分别制定性能基线，不能只报告一个整体平均值。

## 4. 响应耗时指标

### 4.1 Time to First Token

从用户提交问题到看到第一个有效回复 Token 的时间：

```text
TTFT = 第一个输出 Token 时间 - 请求进入系统时间
```

TTFT 直接影响用户对系统“是否卡住”的感受。

如果当前系统不是流式输出，可以记录首字节时间，并在后续改造为流式输出后重新建立基线。

### 4.2 End-to-End Latency

```text
E2E Latency = 最终回复完成时间 - 请求进入系统时间
```

需要分别报告：

- P50：50% 请求低于该时间
- P90：90% 请求低于该时间
- P95：95% 请求低于该时间
- P99：99% 请求低于该时间
- Max：最大耗时

不能只报告平均值，因为少量极慢请求会被平均值掩盖。

### 4.3 Stage Latency

分别统计各阶段 P50、P95 和 P99：

| 阶段 | P50 | P95 | P99 |
|---|---:|---:|---:|
| 意图识别 |  |  |  |
| Embedding |  |  |  |
| ES 检索 |  |  |  |
| Rerank |  |  |  |
| 业务工具 |  |  |  |
| 回答生成 |  |  |  |

### 4.4 Timeout Rate

```text
Timeout Rate = 超时请求数 / 总请求数
```

应区分：

- LLM 超时
- Elasticsearch 超时
- 业务工具超时
- 系统总超时

### 4.5 Slow Request Rate

```text
Slow Request Rate
= 超过该路由 SLO 的请求数 / 该路由总请求数
```

## 5. 冷启动与模型加载

当前项目使用本地 BGE 模型，必须区分冷启动和热运行。

### 5.1 Cold Start Latency

测量进程首次启动后：

- BGE 模型加载时间
- 第一次向量编码时间
- 第一次 Elasticsearch 请求时间
- 第一次 DeepSeek 请求时间

### 5.2 Warm Request Latency

模型、连接和缓存均已就绪时的请求耗时。

### 5.3 Model Reload Count

统计模型是否被重复初始化。正常情况下，Embedding 模型应在进程启动时加载一次并复用。

```text
Model Reload Rate
= 请求过程中发生模型重新加载的次数 / 总请求数
```

如果每个请求都重新创建 BGE Encoder，会明显提高延迟和 CPU、内存开销，应在性能报告中单独暴露。

## 6. 并发与稳定性测试

### 6.1 并发等级

建议至少测试：

```text
1、5、10、20、50 个并发请求
```

根据本地机器或部署资源调整最大并发。

### 6.2 测试类型

#### 基准测试

单用户顺序执行，建立无竞争情况下的延迟基线。

#### 并发测试

在固定并发下持续发送请求，观察吞吐量和 P95 延迟。

#### 突发测试

短时间内从低流量突然增加到高流量，观察排队、超时和恢复能力。

#### 稳定性测试

持续运行 1～4 小时，观察内存增长、连接泄漏、错误率和性能漂移。

### 6.3 核心指标

- Requests Per Second（RPS）
- 成功请求数/分钟
- 并发请求数
- 队列等待时间
- 错误率
- 超时率
- 重试率
- 内存峰值
- CPU 峰值
- Elasticsearch 连接数
- LLM API 限流次数

### 6.4 Throughput

```text
Throughput = 指定时间内成功完成的请求数 / 时间
```

吞吐量必须与任务成功率共同报告，不能把大量快速失败请求计算为有效吞吐。

## 7. 降级与故障场景

需要测试以下故障：

- DeepSeek API 超时
- DeepSeek API 限流
- Elasticsearch 不可用
- Embedding 模型加载失败
- 业务工具不可用
- 单个工具返回异常数据
- 网络短暂中断
- 多个下游服务同时异常

### 7.1 Graceful Degradation Rate

```text
Graceful Degradation Rate
= 故障时成功返回安全、可理解降级结果的请求数
  / 故障测试请求数
```

合格降级包括：

- 告知用户暂时无法查询，而不是编造结果
- 保存用户诉求并转人工
- 检索不可用时不基于记忆猜测政策
- 工具失败时不声称操作已完成
- 保留 `trace_id` 供后续排查

### 7.2 Recovery Time

```text
Recovery Time
= 服务恢复后，Agent 恢复正常成功率所需时间
```

## 8. Token 使用评测

### 8.1 Token 分类

分别记录：

- 意图识别输入 Token
- 意图识别输出 Token
- 回答生成输入 Token
- 回答生成输出 Token
- 历史对话 Token
- 检索上下文 Token
- 工具结果 Token
- 重试产生的额外 Token

### 8.2 Tokens Per Request

```text
Tokens Per Request
= 单次请求所有 LLM 调用 Token 总和
```

### 8.3 Tokens Per Conversation

```text
Tokens Per Conversation
= 同一会话内全部请求 Token 总和
```

### 8.4 Tokens Per Successful Task

```text
Tokens Per Successful Task
= 总 Token 消耗 / 成功完成的任务数
```

相比单次请求 Token，该指标更能反映完成真实用户任务的成本。

### 8.5 Context Utilization

检查输入 Token 中真正被回答使用的上下文比例：

- 是否传入过多历史对话
- Top-K 是否过大
- 是否包含重复 Chunk
- 是否把完整工具原始响应全部传给模型
- Prompt 中是否重复定义相同规则

可以统计：

```text
Context Redundancy Rate
= 重复或与回答无关的上下文 Token / 上下文总 Token
```

## 9. 成本评测

### 9.1 单次请求成本

根据实际模型价格计算：

```text
Request Cost
= 输入 Token × 输入单价
 + 输出 Token × 输出单价
 + Embedding 成本
 + Reranker 成本
 + 其他外部服务成本
```

本地 BGE 无外部 Token 费用，但应记录 CPU、内存和实例资源成本。

### 9.2 单会话成本

```text
Conversation Cost
= 同一会话内所有模型和工具调用成本之和
```

### 9.3 单成功任务成本

```text
Cost Per Successful Task
= 全部请求成本 / 成功任务数
```

失败重试、无效工具调用和过度澄清都会提高该指标。

### 9.4 成本分位数

建议报告：

- 平均成本
- P50 成本
- P95 成本
- 最大单次请求成本
- 最大单会话成本

### 9.5 成本预算命中率

```text
Budget Compliance Rate
= 成本未超过场景预算的请求数 / 总请求数
```

不同路由应设置不同预算，例如 Hybrid 路由通常高于 Clarify 路由。

## 10. 成本优化对照实验

至少对比以下方案：

| 实验 | 优化方式 | 观察指标 |
|---|---|---|
| A | 当前基线 | 质量、延迟、Token、成本 |
| B | 缩短意图 Prompt | 意图 F1、Token、延迟 |
| C | 压缩历史对话 | 多轮准确率、Token |
| D | Top-K 由 5 调整为 3 | 忠实度、完整性、Token |
| E | 检索结果去重 | 幻觉率、Token、相关性 |
| F | 缓存 Embedding 和常见回答 | 命中率、延迟、成本 |
| G | 小模型处理意图，大模型生成答案 | 整体质量、成本 |

任何成本优化都必须同时运行质量回归，不能以牺牲意图准确率、检索 Recall 或回答忠实度为代价。

## 11. 人工转接触发标准

建议明确以下转人工条件：

### 11.1 用户主动要求

- “转人工”
- “我要找真人客服”
- 用户明确拒绝继续与机器人沟通

### 11.2 低置信度

- 意图识别置信度低于阈值
- 多意图冲突且无法通过一次澄清解决
- 关键参数无法确认

### 11.3 高风险业务

- 严重过敏或身体不适
- 高金额退款或争议
- 重复质量投诉
- 用户声称将投诉监管机构
- 涉及欺诈、账户安全或异常支付
- 自动工具无权限处理的操作

### 11.4 自动处理失败

- 工具连续失败或超时
- 业务状态冲突
- 相同问题多轮未解决
- RAG 无可靠知识依据
- 系统降级后仍无法安全回答

### 11.5 情绪与投诉升级

- 用户持续强烈负面情绪
- Agent 连续两次未解决用户问题
- 用户明确表示不接受当前处理方案

## 13. 人工转接核心指标

### 13.1 Handoff Precision

```text
Handoff Precision
= 正确转人工的样本数 / 全部转人工样本数
```

Precision 低说明 Agent 过度转人工，降低自动化价值。

### 13.2 Handoff Recall

```text
Handoff Recall
= 正确转人工的样本数 / 应转人工样本数
```

Recall 低说明高风险或无法处理的请求被错误留在自动流程中。

### 13.3 Missed Handoff Rate

```text
Missed Handoff Rate
= 应转人工但未转的样本数 / 应转人工样本数
```

高风险场景应单独计算 `Critical Missed Handoff Rate`。

### 13.4 Over-handoff Rate

```text
Over-handoff Rate
= 本可自动解决却转人工的样本数 / 可自动解决样本数
```

### 13.5 Eligible Automation Resolution Rate

只在“适合自动处理”的任务中计算自动解决率：

```text
Eligible Automation Resolution Rate
= 自动成功解决的适合自动处理任务数
  / 全部适合自动处理任务数
```

不能把必须人工处理的高风险任务纳入分母来追求虚假的自动解决率。

## 14. 转接质量评测

成功触发转人工不代表转接质量合格。

### 14.1 Handoff Success Rate

```text
Handoff Success Rate
= 成功创建人工工单并返回有效接入状态的次数
  / 发起人工转接次数
```

### 14.2 Summary Completeness

```text
Summary Completeness
= 转接摘要已覆盖的必要字段数 / 全部必要字段数
```

转接摘要建议包含：

- 用户核心诉求
- 用户情绪和紧急程度
- 订单号及相关商品
- 已识别意图
- Agent 已执行的检索和工具操作
- 工具返回结果
- 已向用户确认的信息
- 当前失败原因
- 建议人工下一步处理

### 14.3 Context Preservation Rate

```text
Context Preservation Rate
= 人工无需重新询问的已知关键信息数
  / 用户已经提供的关键信息数
```

### 14.4 Repeat Information Rate

```text
Repeat Information Rate
= 转人工后用户被要求重复提供的信息数
  / 转接前已经提供的信息数
```

### 14.5 Handoff Latency

从 Agent 决定转人工到工单创建成功或人工接入的时间。

分别报告：

- 工单创建 P50/P95
- 排队时间 P50/P95
- 高优先级任务等待时间

## 15. 人工转接结果反馈

成熟 Agent 应将人工处理结果反馈到评测和知识体系中。

建议记录：

- 人工是否认可转接必要性
- Agent 摘要是否准确
- 人工是否修改了意图标签
- 人工最终采用的处理方案
- 是否发现知识库缺口
- 是否发现工具能力缺口
- 用户是否需要重复描述问题
- 最终是否解决

这些数据可以用于优化：

- 意图识别阈值
- 转人工规则
- FAQ 和政策知识库
- 工具覆盖范围
- 高风险场景处理流程

## 17. 建议验收标准

以下标准适合作为本地原型和面试项目的阶段目标，不代表当前系统已经达到。正式生产环境应根据业务流量和模型服务能力重新制定 SLO。

### 17.1 性能与稳定性

| 指标 | 合格线 | 较好目标 |
|---|---:|---:|
| Clarify 路由 P95 | ≤ 4s | ≤ 2.5s |
| Knowledge 路由 P95 | ≤ 8s | ≤ 5s |
| Business API 路由 P95 | ≤ 7s | ≤ 4s |
| Hybrid 路由 P95 | ≤ 10s | ≤ 7s |
| ES 检索 P95 | ≤ 500ms | ≤ 200ms |
| 请求成功率 | 97% | ≥ 99% |
| 系统超时率 | ≤ 2% | ≤ 0.5% |
| 降级成功率 | 90% | ≥ 98% |

### 17.2 成本

成本目标应根据实际模型单价确定，至少满足：

| 指标 | 建议目标 |
|---|---:|
| Token 统计覆盖率 | 100% |
| 成本统计覆盖率 | 100% |
| 超预算请求率 | ≤ 5% |
| 无效重试 Token 占比 | ≤ 2% |
| 重复上下文 Token 占比 | ≤ 10% |

### 17.3 人工转接

| 指标 | 合格线 | 较好目标 |
|---|---:|---:|
| Handoff Precision | 85% | ≥ 92% |
| Handoff Recall | 90% | ≥ 97% |
| Critical Missed Handoff Rate | ≤ 1% | 0 |
| Over-handoff Rate | ≤ 15% | ≤ 8% |
| Handoff Success Rate | 97% | ≥ 99% |
| Summary Completeness | 90% | ≥ 97% |
| Context Preservation Rate | 90% | ≥ 98% |
| Repeat Information Rate | ≤ 10% | ≤ 3% |

```json
  "error_type": null
}
```

## 19. 错误类型

建议为异常请求增加以下标签：

- `intent_latency`：意图识别阶段过慢
- `embedding_cold_start`：Embedding 冷启动耗时过长
- `embedding_reloaded`：请求内重复加载模型
- `retrieval_latency`：检索延迟超标
- `tool_latency`：业务工具响应过慢
- `generation_latency`：回答生成延迟超标
- `queue_timeout`：排队导致超时
- `llm_rate_limit`：模型 API 限流
- `service_unavailable`：依赖服务不可用
- `degradation_failed`：故障时未安全降级
- `token_over_budget`：Token 超过预算
- `context_redundancy`：历史或检索上下文重复过多
- `unnecessary_llm_call`：发生不必要的模型调用
- `missed_handoff`：应该转人工却未转
- `over_handoff`：本可自动处理却转人工
- `handoff_summary_missing`：转接摘要缺少关键信息
- `context_lost`：人工接手后丢失已有信息
- `handoff_failed`：人工工单创建失败

## 20. 评测执行流程

1. 为 Agent 全链路增加 `trace_id` 和阶段级日志。
2. 冻结模型、Prompt、检索配置、工具版本和硬件环境。
3. 构建不同路由、不同长度和不同风险等级的性能测试集。
4. 先进行单并发基线测试，区分冷启动与热请求。
5. 逐步进行 5、10、20、50 并发测试。
6. 执行突发流量、依赖故障和长时间稳定性测试。
7. 统计 TTFT、E2E 延迟、吞吐量、超时率和资源使用。
8. 记录每个 LLM 阶段的输入输出 Token 及实际费用。
9. 构建人工转接标注集，计算 Precision、Recall 和摘要质量。
10. 对延迟瓶颈、超预算请求和漏转人工案例进行根因分析。
11. 优化后运行质量回归，确认性能和成本优化没有损害任务成功率。
12. 保存完整实验配置和结果，形成可复现的对照报告。
