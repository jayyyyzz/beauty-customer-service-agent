# 美妆电商客服 Agent——性能、成本与人工转接专项测评报告

测评日期：2026-07-16  
结论：**当前版本未通过专项验收。成本较低且业务工具性能良好，但知识类链路延迟和人工转接召回率不达标。**

## 1. 测评范围与环境

本次使用真实 DeepSeek API、真实 Elasticsearch 索引和本地 SQLite 业务工具执行测评，覆盖：

- 冷启动、热运行、单并发和 4 并发
- `knowledge_base`、`business_api`、`hybrid`、`clarify`、`fallback`、`human_handoff`
- 阶段级耗时、Token、费用和缓存命中
- 24 条人工转接专项样本
- Elasticsearch、业务工具、LLM 超时和紧急安全路径故障注入
- 关闭 LLM Rerank、BM25 快速路径的优化对照实验

测试环境：

| 项目 | 配置 |
|---|---|
| 操作系统 | Windows 11 家庭中文版 10.0.26200 |
| CPU | Intel Core i7-8750H，6 核 12 线程，2.20GHz |
| 内存 | 15.88GB；测评后可用约 2.3GB |
| Python | 3.14.5 |
| Elasticsearch | 8.17.4，Docker 单节点，1GB JVM Heap |
| ES 索引 | `customer_service_knowledge_v1`，实际 25,520 文档，约 184MB；当前知识源应为 26,196 文档 |
| ES 测评后资源 | CPU 12.33%，内存 1.694GiB |
| LLM | `deepseek-chat`，非流式 ChatCompletions |
| 最大 Agent 并发 | 4 |

执行前发现 `agent_pipeline.py` 仍引用已改名的旧意图 Prompt 目录，导致 5 项测试无法导入。修复后完成最终回归，**45 项自动化测试全部通过**。

环境检查同时发现 ES 索引比当前知识源少 676 条。尝试执行增量同步，但本地纯 NumPy 数据准备/向量构建在 15 分钟内未完成，索引数量仍为 25,520。本报告的在线检索结果基于该实际索引，不能代表完整知识源覆盖率。

## 2. 性能结果

### 2.1 端到端延迟

共执行 19 个性能请求，成功 19 个，样本内成功率 100%。

| 指标 | 结果 |
|---|---:|
| E2E P50 | 12.98s |
| E2E P90 | 19.59s |
| E2E P95 | 21.36s |
| E2E P99 | 23.08s |
| 最大值 | 23.51s |
| 平均值 | 11.59s |

各路由结果：

| 路由 | 样本数 | P50 | P95 | 方案合格线 | 结论 |
|---|---:|---:|---:|---:|---|
| `knowledge_base` | 4 | 17.91s | 19.09s | ≤8s | 不通过 |
| `business_api` | 3 | 17.17s | 17.87s | ≤7s | 不通过 |
| `hybrid` | 2 | 18.24s | 22.98s | ≤10s | 不通过 |
| `clarify` | 1 | 16.48s | 16.48s | ≤4s | 不通过，样本较少 |
| `fallback` | 3 | 4.97s | 10.77s | 未设定 | 长尾明显 |
| `human_handoff` | 5 | 0.013s | 6.43s | 未设定 | 紧急规则路径很快，低置信度路径仍调用 LLM |

`business_api` 最终路由中出现了 Embedding/RAG 阶段，说明多意图识别会把部分纯订单请求扩展为额外知识任务。路由表中的延迟是最终用户体验，不等同于业务工具自身耗时。

### 2.2 阶段耗时

| 阶段 | 调用数 | P50 | P95 | 结论 |
|---|---:|---:|---:|---|
| 意图识别 | 22 | 1.58s | 10.36s | P95 长尾严重，且存在二次识别 |
| Embedding | 11 | 0.01ms | 3.80s | 重复查询缓存有效，新查询仍需约 2.3～3.8s |
| ES 检索/RRF/MMR | 11 | 4.28s | 5.25s | 远高于 500ms 合格线 |
| LLM Rerank | 11 | 2.86s | 8.01s | 主要长尾与 Token 消耗源之一 |
| 回答生成 | 13 | 3.30s | 6.99s | 需要流式输出和上下文压缩 |
| 业务工具 | 7 | 1.10ms | 2.32ms | 性能良好，不是瓶颈 |

冷启动知识请求为 19.21s，其中首次 Embedding 为 3.78s；相同查询热运行后为 10.71s，降低约 44.2%。但是缓存只对相同问题生效，新问题编码仍为秒级。

### 2.3 并发与吞吐

- 单并发热运行：6 个请求耗时 54.09s，约 0.111 RPS。
- 4 并发：12 个请求耗时 40.95s，吞吐为 **0.293 RPS**。
- 4 并发相对单并发吞吐提高约 164%，但请求 P95 仍超过 20s。

本次未执行 10、20、50 并发和 1～4 小时稳定性测试。当前 Agent 最大并发配置为 4，且宿主机剩余内存较低；继续提高并发前应先降低单请求 CPU 和检索开销。

### 2.4 TTFT

**未测得。** 当前使用非流式 ChatCompletions，只能记录完整模型调用耗时，无法取得首 Token 时间。该项不满足方案中的 TTFT 可观测性要求。

## 4. 人工转接结果

测试集共 24 条，其中应转人工 12 条、不应立即转人工 12 条。

| 指标 | 结果 | 合格线 | 结论 |
|---|---:|---:|---|
| Handoff Precision | 83.33% | ≥85% | 不通过 |
| Handoff Recall | 41.67% | ≥90% | 严重不通过 |
| Missed Handoff Rate | 58.33% | ≤10% | 严重不通过 |
| Over-handoff Rate | 8.33% | ≤15% | 通过 |
| Critical Missed Handoff Rate | 25.00% | ≤1% | 严重不通过 |
| Handoff Success Rate | 100% | ≥97% | 通过 |

漏转类型：

- 用户明确要求人工：3/3 漏转，均进入 `fallback`。
- 投诉监管升级：漏转。
- 账户被盗、异常支付：漏转，构成关键风险漏转。
- 多轮查询失败、用户拒绝重复信息：漏转。
- 高金额退款争议：漏转。
- 严重和普通护肤不良反应：5/5 正确转接。

误转类型：

- “还有别的办法吗？”因模型低置信度直接转人工，属于一次过度转接。

成功创建的 6 张工单均成功写入 SQLite。200 次本地工单创建微基准：P50 9.14ms、P95 13.23ms、P99 19.56ms，工单存储性能合格。

在“已经成功创建的工单”中，自动字段检查得到摘要完整率和上下文事实保留率 100%；但这个结果只覆盖成功转接样本。把 7 个漏转样本计入服务闭环后，有效转接信息覆盖率最多为 41.67%。此外，本次没有真实人工客服接手，无法测量人工认可率、排队时间和用户重复信息率。

## 5. 故障与降级

| 故障 | 结果 |
|---|---|
| Elasticsearch 不可用 | 返回安全的通用失败提示，但未自动创建人工工单 |
| 业务工具不可用 | 返回安全的通用失败提示，但未保存待办或自动转人工 |
| LLM 超时 | 异常上抛，直接调用链没有安全降级 |
| 紧急护肤风险且 LLM 不可用 | 规则路径正常返回就医提示并创建工单 |

按“是否返回安全且可理解的结果”计算，降级成功率为 **75%**；如果要求同时保存诉求并创建人工待办，则只有紧急安全路径形成完整闭环。

## 6. 优化对照实验

对 3 个知识类问题各重复 2 次，使用真实 LLM 和 ES：

| 方案 | 平均延迟 | 平均 Token | 平均成本 | 非空召回率 |
|---|---:|---:|---:|---:|
| A：RRF+MMR+LLM Rerank | 12.25s | 3,883 | $0.00025748 | 100% |
| B：RRF+MMR，关闭 LLM Rerank | 7.94s | 1,028 | $0.00015259 | 100% |
| C：BM25，关闭 LLM Rerank | 6.24s | 1,029 | $0.00015035 | 100% |

相对基线：

- 关闭 LLM Rerank：延迟下降 35.2%，Token 下降 73.5%，成本下降 40.7%。
- BM25 快速路径：延迟下降 49.1%，Token 下降 73.5%，成本下降 41.6%。

该对照样本只有 6 次/方案，且只检查非空召回和人工查看结果，不能证明完整检索质量不下降。调整生产默认值前必须在 65 条检索基准上回归 Recall@K、MRR、NDCG 和答案忠实度。

## 8. 测评限制

- 性能样本 19 条，路由分组样本较少，分位数用于发现方向性问题，不等同于生产压测。
- 只测试到 4 并发，未执行 10/20/50 并发和 1～4 小时稳定性测试。
- 当前非流式输出，无法测量 TTFT。
- 未接入真实人工排队系统，无法测量排队时间、重复询问信息和人工认可率。
- 优化对照只有 3 个问题、每个方案 2 次；正式切换检索方案前必须做完整质量回归。
- 测评索引比当前知识源少 676 条；知识覆盖和检索质量结果存在下偏风险。
- DeepSeek 官方页面显示 `deepseek-chat` 将于 2026-07-24 15:59 UTC 弃用，报告中的模型别名和价格需要在迁移后重新确认。
