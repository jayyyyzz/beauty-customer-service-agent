# 性能、成本与人工转接优化建议

## P0：上线或对外演示前必须完成

### 1. 增加确定性的人工转接策略层

不要只依赖模型置信度和护肤症状词。应在意图识别前增加规则策略，至少覆盖：

- 用户明确要求人工或拒绝继续与机器人沟通
- 账户被盗、异常支付、欺诈和身份安全
- 投诉消协、监管机构、媒体或法律升级
- 高金额退款、状态冲突和责任争议
- 相同工具连续失败、同一问题连续两轮未解决
- 严重不良反应及需要人工跟进的普通过敏售后

建议统一输出：`should_handoff / reason / priority / signals / required_summary_fields`。目标是将 Recall 从 41.67% 提升至 90%以上，并把关键漏转率降为 0。

### 2. 修正护肤安全规则的误触发

当前只要出现“泛红、爆痘、脱皮”等词就要求转人工，容易把“平时容易泛红”“我属于敏感肌”误判为产品不良反应。应加入因果和时间条件，例如同时出现：

- “使用/涂/敷/刚用”
- “之后/后/开始”
- 症状词和严重度词

普通肤质描述只进入安全提示或知识咨询，不直接转人工。

### 3. 为 LLM、ES 和业务工具增加完整降级闭环

当前 LLM 超时会直接失败，ES 和工具故障只返回通用提示。应增加：

- 分阶段超时，而不是只依赖 90 秒总超时
- 指数退避和最多一次受控重试
- Circuit Breaker，避免故障期间持续压垮下游
- 自动创建人工工单，保存用户诉求、已知槽位、失败阶段和 `trace_id`
- 工具失败时禁止生成“已完成”表述

### 4. 立即迁移模型别名

DeepSeek 官方页面显示 `deepseek-chat` 将于 **2026-07-24 15:59 UTC** 弃用，并映射到 V4 Flash 非思考模式。应改为显式模型名，重新执行意图、回答、性能和成本回归，避免临近面试或演示时接口失效。

## P1：性能和成本核心优化

### 5. 将 LLM Rerank 改为条件触发

对照实验显示，关闭 LLM Rerank 后：

- 平均延迟从 12.25s 降至 7.94s，下降 35.2%
- 平均 Token 从 3,883 降至 1,028，下降 73.5%
- 平均成本下降 40.7%

建议默认使用 RRF+MMR，只在以下情况启用 LLM Rerank：

- Top1/Top2 分数接近且候选主题冲突
- 高风险售后、政策或安全问题
- 检索置信度低于阈值
- 多意图请求需要跨资料合并

切换前必须跑完整检索和回答质量回归。

### 6. 增加 BM25 快速路径

BM25 无 Rerank 方案平均延迟 6.24s，比基线下降 49.1%。建议对以下问题使用快速路径：

- 订单号、快递单号、品牌名、商品型号等精确实体
- 发票、保质期、固定政策等高关键词匹配问题
- 已知 FAQ 标准问法

口语化、近义表达、肤质和功效咨询继续使用向量或混合检索。路由方式应由离线 Recall@K 数据决定，不能只看延迟。

### 7. 替换纯 NumPy BGE 推理

新问题 Embedding 实测约 2.3～3.8s，缓存命中才接近 0ms。建议优先级：

1. 使用 ONNX Runtime/OpenVINO 或 `sentence-transformers` 优化推理。
2. 进程启动时预热模型和常见查询。
3. 保留 LRU 查询缓存，并增加命中率指标。
4. 多意图任务共享同一用户 Query 向量，避免重复编码。
5. 有 GPU 时单独部署 Embedding 服务并支持批量编码。

目标：新查询 Embedding P95 降至 200ms 以内。

### 8. 优化 ES RRF/MMR 链路

ES 检索 P95 为 5.25s，目标是 500ms。建议：

- 复用 HTTP 连接，避免每次使用 `urllib` 新建连接和 SSL Context
- 使用 `_msearch` 或 ES 原生 RRF，减少两路 HTTP 往返
- 降低 `candidate_k=30` 和 `num_candidates=200`，用质量曲线选最小可用值
- 不要把 30～60 条 512 维向量完整返回 Python 后做纯 Python MMR
- 将 MMR/去重迁移到向量化实现，或只对小候选集执行
- 调整索引映射与 `_source` 字段，减少响应体
- 在内存更充足的环境重新测量；当前宿主机只剩约 2.3GB 可用内存

### 9. 减少意图识别调用和 Prompt

19 个请求发生 22 次意图识别，意图阶段消耗总 Token 的 42.96%，P95 达 10.36s。建议：

- 合并单意图与多意图识别，禁止无必要的第二次 LLM 判断
- 对“转人工、订单号、物流、开票”等强规则场景先走规则分类
- 缩短三级分类定义，只把一级路由和必要子类交给在线模型
- 使用小模型或本地分类器负责一级路由，大模型只处理歧义样本
- 使用标定后的置信度，不直接信任模型自报分数

### 10. 重构知识索引增量发布链路

环境检查显示知识源应有 26,196 条，ES 实际只有 25,520 条；增量同步运行 15 分钟仍未完成且索引数未变化。建议：

- 在数据清洗阶段生成持久化 `document_id/content_hash/embedding_version` Manifest，避免每次发布重新扫描和构造全部文档
- 将“比较变更集”和“生成向量”拆成两个可观测阶段，先快速输出计划和预计耗时
- 使用 ONNX/OpenVINO 批量编码新增文档，避免纯 NumPy 长文本批处理
- 每批写入后持久化 checkpoint，支持中断恢复
- 发布完成后强制校验源文档数、索引文档数、失败文档数和删除文档数
- 为增量发布设置 SLO，例如 1,000 条变更在 5 分钟内完成

## P2：体验与运营闭环

### 11. 支持流式输出并记录 TTFT

当前无法测量 TTFT。应改为流式 ChatCompletions，记录：

- 请求进入时间
- 首 Token 时间
- 完整生成时间
- 用户取消时间

即使完整答案仍需数秒，也应优先把 TTFT 控制到 2.5 秒以内。

### 12. 完善人工转接摘要

工单摘要应结构化包含：

- 用户诉求与情绪/紧急程度
- 订单号、商品和已知槽位
- 历史中已提供的信息
- 已识别意图、检索来源和工具操作
- 工具返回结果与失败原因
- Agent 已经向用户确认的内容
- 建议人工下一步动作

当前成功工单能保存问题和安全上下文，但漏转样本没有任何工单。应以“全部应转样本”为分母计算有效摘要覆盖率。

### 13. 建立更完整的专项回归集

当前 24 条转接样本适合发现主要缺口，但正式验收应扩充到 200～400 条，覆盖：

- 用户主动要求人工的多种表达
- 投诉升级、欺诈、异常支付和高金额争议
- 工具失败、状态冲突和多轮未解决
- 高风险与普通护肤不良反应边界
- 强情绪但仍可自动解决的反例

需要人工客服参与标注和复核，补齐人工认可率、重复信息率和真实排队时间。

### 14. 设置路由级预算和告警

当前平均成本约 $0.000188/请求，绝对值较低。建议仍为不同路由设置预算，例如：

- Clarify/Fallback：最多 1 次短 LLM 调用
- Business API：意图+答案，不允许无关 RAG/Rerank
- Knowledge：限制 Top-K 和上下文字符数
- Hybrid/Multi-intent：设置更高预算并监控异常长尾

当 Token、延迟或重试超过预算时，记录 `token_over_budget` 并触发降级，而不是继续无界追加上下文。

## 建议实施顺序

1. 人工转接规则与关键风险兜底。
2. LLM 超时/ES/工具失败自动工单闭环。
3. 迁移 DeepSeek 显式模型名。
4. 条件化 LLM Rerank，并做质量回归。
5. ONNX/OpenVINO Embedding 和 ES 检索优化。
6. 重构知识索引增量发布和一致性校验。
7. 流式输出、TTFT 和队列指标。
8. 200～400 条人工转接回归集与真实人工反馈。
