# 优化后剩余建议

当前专项性能目标已全部通过。后续优化重点不再是修复明确的 SLO 失败，而是提升用户感知、长期稳定性和生产可信度。

## P0：补齐生产级响应体验与稳定性

### 1. 流式输出并记录 TTFT

当前 Answer Generation P95 为 5.446s，完整响应已达标，但 Agent 仍采用非流式输出，无法测量首 Token 时间。建议接入流式响应，并把 TTFT、完整生成耗时和中断率分别纳入指标，目标 TTFT ≤2.5s。

### 2. 执行长稳与高并发压测

完成 10/20/50 并发以及 1～4 小时稳定性测试，重点观察：

- Elasticsearch、模型客户端和业务data连接数。
- 容器内存、JVM Heap、进程 RSS 和退出码 137 风险。
- 限流、队列等待、超时、重试和恢复时间。
- P95/P99 是否随运行时间持续抬升。

### 3. 为连接池增加生命周期与健康治理

当前查询和入库已使用持久连接池。建议在应用关闭时统一释放连接，并增加连接池错误率、活动连接数、ES `took`、网络耗时和重连次数监控。

## P1：继续降低模型与向量开销

### 4. 高频 FAQ 采用证据模板或版本化缓存

护肤步骤、保质期查看方式、基础肤质适配等高频问题，可在证据充分时使用确定性模板或答案缓存。缓存键应包含标准问题、知识版本、安全策略版本和模型版本，避免旧答案污染。

### 5. 优化新查询 Embedding

Embedding P95 为 0.653s，虽然已不影响当前 SLO，但仍是冷查询的固定成本。可评估 ONNX Runtime/OpenVINO、进程启动预热和批量并行编码，同时保留现有 LRU 缓存。

### 6. 评估异步 ES 客户端或 `_msearch`

当前 ES 检索 P95 已达到 0.438s。若并发进一步提升，可比较异步客户端、`_msearch` 或 ES 原生 RRF，减少线程池切换和双路召回的连接竞争。

## P2：扩大质量与运营验证

### 7. 运行完整检索质量回归

在完整检索基准上重新计算 Recall@3、MRR、NDCG、重复率和无答案识别，确认关闭默认 Rerank、缩小候选集以及索引元数据同步没有损害复杂问题召回。

### 8. 扩充人工转接测试集

从 24 条扩充至 200～400 条，并由人工客服复核转接必要性、优先级、摘要完整度和上下文保留情况。

### 9. 接入真实人工队列指标

接入工单系统后测量排队时间、人工接入成功率、重复询问信息率、首次解决率和最终解决率，形成“自动处理—转人工—闭环解决”的完整观测链路。

### 10. 增加真实流量回放与容量模型

使用脱敏后的真实咨询分布回放，按知识、业务、混合、澄清和人工转接占比计算单位请求成本、峰值容量和每日预算，而不仅依赖均匀构造的专项样本。
